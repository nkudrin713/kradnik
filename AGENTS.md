# Agent Instructions

## Main Mode

### Communication

- Отвечай на языке пользователя.
- Стиль: кратко, сухо, технически.
- Можно сжатые фразы, но без потери смысла.
- Сохраняй объект, действие и причину.
- Не превращай речь в телеграфный бред.
- Не используй случайный набор слов и дубли слов.
- Без вступлений, вежливых оборотов, воды и длинных объяснений.
- Не пиши: `конечно`, `давай`, `хорошо`, `понял`, если это не нужно для смысла.
- Не описывай очевидные действия.
- Не объясняй, что следуешь AGENTS.md.
- Не раскрывай ход мыслей.
- Не показывай sandbox-ограничения, если они не критичны.

### Answer Format

- Простая задача: 1–2 короткие фразы.
- Ошибка: `Причина → фикс`.
- Код изменен: список файлов и действий.
- Тесты: одна строка.
- Риск: одна строка, только если есть.
- Нет изменений: `Готово. Изменений нет.`

Пример:

```text
Причина найдена: duration берется из metadata, но дальше не передается.
Фикс: пробросить duration в actual-вызов.
Изменено: DownloadService.kt — передача duration исправлена.
Тесты: ./gradlew test --tests DownloadServiceTest
Риск: низкий. Затронут только расчет duration.
```

Плохо:

```text
Расчет: секунд секунд с метаданных не проп?
Смотрю actual вызов.
duration metadata actual проброс не.
```

## Work Modes

### Small Fix

Для простой локальной правки:

1. Найти причину.
2. Исправить минимально.
3. Запустить самый узкий релевантный тест, если есть.
4. Ответить по формату.

### Investigation

Если причина неясна:

1. Сначала собрать факты.
2. Не менять код до рабочей гипотезы.
3. Дать краткий вывод: причина / варианты / следующий шаг.
4. Если нужна правка — сделать минимально достаточную.

### Broad Task

Если пользователь дал широкое разрешение вроде `делай всё подряд`:

1. Выполняй задачи по порядку.
2. Делай отдельный коммит на каждую законченную задачу, если пользователь этого просил.
3. После каждого коммита продолжай.
4. Останавливайся только на ошибке, конфликте, секрете или destructive operation.

### Deploy / CI / Server

Для деплоя, CI и сервера:

1. Проверить ветку и `git status --short`.
2. Проверить целевое окружение: `prod/main` или `test/develop`.
3. Сначала смотреть реальные логи и факты, потом менять код или конфиги.
4. После деплоя проверить commit/image/container/logs.
5. Если деплой касается Telegram-бота, учитывать два окружения и раздельные базы.

## Token Economy

- Не вставлять большие файлы, логи и diff целиком.
- Показывать только релевантные строки ошибки.
- Для поиска использовать `rg` / `fd`, не читать весь проект без нужды.
- Открывать только файлы, нужные для задачи.
- Для планов максимум 3–5 пунктов.
- Не объяснять базовые вещи, если пользователь не просил.
- Не повторять уже сказанное.
- Если команда очевидна по контексту, не расписывать ее назначение.

## Code Changes

- Делай только прямой запрос пользователя.
- Не меняй лишнее.
- Не трогай соседний код.
- Не улучшай стиль без запроса.
- Не рефактори без запроса.
- Не меняй поведение без запроса.
- Следуй текущим паттернам проекта.
- Хочешь сделать незапрошенное изменение — сначала спроси.
- Изменения структуры базы данных и таблиц делаются только через Flyway-миграции.
- Запрещено менять V1 init migration без прямого запроса.

## Git Safety

- Любые локальные изменения считать пользовательскими.
- Перед изменениями смотреть `git status --short`.
- Чужие незакоммиченные изменения не трогать.
- Не делать `git reset --hard`, `git clean`, `git checkout -- .`, `git restore .` без прямого запроса.
- Не переписывать историю без прямого запроса.
- Если есть готовый пользовательский коммит, не менять его без запроса.
- Коммитить только по прямому запросу.
- Если пользователь просит отдельные коммиты: одна задача = один коммит.
- Перед коммитом проверить `git status --short` и релевантный diff.
- В коммит не добавлять чужие незакоммиченные изменения, если пользователь не сказал `коммить всё`.
- После пуша сообщить branch и commit hash.

## Destructive Operations

Destructive operation: удаление данных, контейнеров, volumes, secrets, force-push, reset, clean, изменение prod-инфраструктуры.

- Требует отдельного явного разрешения.
- Если разрешение уже дано в текущей задаче — можно выполнять в его границах.
- Если границы неясны — остановиться и коротко назвать риск.

## Dependencies

- Новые зависимости — только при явной нужде.
- Перед добавлением: 1 короткая причина.
- `latest` запрещен.
- Всегда фиксируй конкретную версию.
- Версия: свежая, стабильная.

## Kotlin

- Предпочитай простой Kotlin-код.
- Сохраняй стиль Kotlin/Spring и кодстайл проекта/файла.
- Используй constructor injection.
- Не используй `!!`, если можно нормально обработать nullable.
- Не добавляй `lateinit`, если можно обойтись constructor injection.
- Не злоупотребляй scope-функциями `let`, `also`, `apply`, `run`.
- Wildcard import запрещен. Нельзя: `import something.*`.
- Если передаешь в функцию или конструктор много параметров, всегда указывай их названия.

## Tests

- Сначала запускать самый узкий релевантный тест.
- Полный `./gradlew test` запускать перед коммитом/деплоем или если изменение широкое.
- Для бизнес-логики предпочитать unit-тесты без Spring-контекста.
- Spring-контекст поднимать только для integration/repository/controller tests.
- Для моков в Kotlin предпочтителен MockK.
- Для тестовых данных предпочтительны маленькие fixtures/factories. Используй Arb, если он уже есть в проекте.
- Если используется random в тестах, фиксируй seed или делай генерацию воспроизводимой.
- Если тесты не запускались — указать причину коротко.

## CLI Tools

Используй инструменты, когда они полезны:

- `rg` — быстрый поиск по содержимому. Предпочитай вместо `grep -R`.
- `fd` — быстрый поиск файлов. Предпочитай вместо `find` для простых задач.
- `jq` — просмотр и обработка JSON.
- `yq` — просмотр и обработка YAML, особенно `docker-compose.yml`, GitHub Actions и конфиги.
- `bat` — просмотр файлов с подсветкой.
- `tree` — быстрый обзор структуры проекта.
- `delta` — удобный просмотр git diff.
- `ast-grep` — структурный поиск и правки кода, если regex через `rg` хрупкий.
- `gh` — GitHub Actions, PR, логи и статусы.
- `docker` / `docker compose` — контейнеры, логи и сервисы.
- `psql` / `pg_isready` — PostgreSQL, если доступ настроен.
- `just` — запуск проектных команд, если есть `justfile`.

## Tool Preferences

- Используй `./gradlew`, а не системный `gradle`.
- Используй `docker compose`, а не старый `docker-compose`.
- Используй `gh run view --log-failed` для анализа упавшего CI, если `gh` авторизован.
- Для Docker-диагностики сначала используй `docker compose ps` и `docker compose logs`.
- Для PostgreSQL сначала используй read-only проверки и диагностические запросы.
