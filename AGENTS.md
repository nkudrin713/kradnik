# Agent Instructions

## Main Mode

## Communication

- Отвечай на языке пользователя.
- Экономь токены жестко.
- Пиши максимально коротко.
- Меньше слов.
- Смысл важнее красоты.
- Без воды.
- Без вежливости.
- Без вступлений.
- Без повторов.
- Без длинных планов.
- Без длинных итогов.
- Без лишних объяснений.
- Без советов “на всякий случай”.
- Без фраз “конечно”, “давай”, “хорошо”, “понял”.
- Без эмоций.
- Без украшений текста.
- Без канцелярита.
- Без сложных конструкций.
- Без очевидных вещей.
- Не объясняй базу, если не просили.
- Не расписывай ход мыслей.
- Не описывай каждое действие.
- Не пиши “я сделал”, если и так понятно.
- Можно писать примитивно.
- Можно писать рублено.
- Можно как пещерный человек.
- Главное: понятно, коротко, полезно.

## Answer Format

- Простая задача: 1 фраза.
- Нужны детали: 2–3 короткие фразы.
- Код изменен: короткий список.
- Ошибка: причина → фикс.
- Тесты: одна строка.
- Риск: одна строка, только если есть.
- Нет изменений: `Готово. Изменений нет.`
- Не добавляй финальное резюме, если оно не нужно.

## Style Examples

Плохо:
> Конечно, я могу это сделать. Сейчас я внес изменения и дополнительно объясню, что именно поменялось.

Хорошо:
> Готово.

Плохо:
> Я обнаружил проблему, связанную с тем, что переменная может быть null в этом месте.

Хорошо:
> Причина: возможен null.  
> Фикс: добавить проверку.

Плохо:
> Тесты я не запускал, потому что изменение было небольшим и локальным.

Хорошо:
> Тесты: не запускал. Малое изменение.

Плохо:
> Ниже я приведу пример того, как можно реализовать это поведение в вашем проекте.

Хорошо:
> Пример:

Плохо:
> В целом этот подход является достаточно хорошим, но есть несколько нюансов, которые стоит учитывать.

Хорошо:
> Норм. Есть риски:

## Code Changes

- Делай только прямой запрос.
- Не меняй лишнее.
- Не трогай соседний код.
- Не улучшай стиль без запроса.
- Не рефактори без запроса.
- Не меняй поведение без запроса.
- Следуй текущим паттернам проекта.
- Чужие незакоммиченные изменения не трогай.
- Хочешь сделать незапрошенное изменение — сначала спроси.

## Dependencies

- Новые зависимости — только при явной нужде.
- Перед добавлением: 1 короткая причина.
- `latest` запрещен.
- Всегда фиксируй конкретную версию.
- Версия: свежая, стабильная.

## Kotlin

- Wildcard import запрещен.
- Нельзя: `import something.*`.
- Только конкретные импорты.
- Стиль Kotlin/Spring сохранять.

## Tests And Build

- Не запускай тесты после каждого шага.
- Малое локальное изменение — тесты не запускать.
- Запускай тесты:
    - перед коммитом;
    - перед пушем;
    - после крупного рефакторинга;
    - когда много изменений в git index.
- Полный прогон:

```bash
./gradlew test
```

- Если тесты не запустились: причина + риск.

## Git

- Не делай коммит без прямого запроса.
- Не делай destructive-команды без прямого разрешения.
- Запрещено без разрешения:
    - `git reset --hard`
    - `git checkout --`
    - массовое удаление файлов
- В конце укажи измененные файлы.

## Final Response

Формат финала:

```text
Изменено:
- file.kt — что сделано

Тесты:
- не запускал / ./gradlew test

Риск:
- коротко, если есть
```

Если изменений нет:

```text
Готово. Изменений нет.
```
