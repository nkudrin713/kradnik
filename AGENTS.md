# Agent Instructions

## Project Context

Проект: Telegram-бот для скачивания медиа по ссылке.

Основной стек:

- Kotlin
- Spring Boot
- PostgreSQL
- Docker / Docker Compose
- Gradle
- yt-dlp
- Telegram Bot API
- GitHub Actions

## Instruction Priority

- Эти инструкции специфичны для текущего репозитория.
- Если есть конфликт с глобальной персонализацией Codex, приоритет у этого `AGENTS.md`.
- Общая коммуникация, экономия токенов и git safety вынесены в персонализацию Codex и не дублируются здесь.

## Deploy / CI / Server

Для деплоя, CI и сервера:

1. Проверить ветку и `git status --short`.
2. Проверить целевое окружение: `prod/main` или `test/develop`.
3. Сначала смотреть реальные логи и факты, потом менять код или конфиги.
4. После деплоя проверить commit/image/container/logs.
5. Если деплой касается Telegram-бота, учитывать два окружения и раздельные базы.

## Code Changes

- Изменения структуры базы данных и таблиц делаются только через Flyway-миграции.
- Запрещено менять V1 init migration без прямого запроса.
- Не меняй тексты пользовательских сообщений без запроса.
- Не меняй формат Telegram-команд без запроса.
- Не меняй публичное поведение бота без запроса.

## Kotlin

- Предпочитай простой Kotlin-код.
- Сохраняй стиль Kotlin/Spring и кодстайл проекта/файла.
- Используй constructor injection.
- Не используй `!!`, если можно нормально обработать nullable.
- Не добавляй `lateinit`, если можно обойтись constructor injection.
- Не злоупотребляй scope-функциями `let`, `also`, `apply`, `run`.
- Wildcard import запрещен. Нельзя: `import something.*`.
- Если передаешь в функцию или конструктор много параметров, всегда указывай их названия.

## Tests

- Для бизнес-логики предпочитать unit-тесты без Spring-контекста.
- Spring-контекст поднимать только для integration/repository/controller tests.
- Для моков в Kotlin предпочтителен MockK.
- Для тестовых данных предпочтительны маленькие fixtures/factories.
- Используй Arb, если он уже есть в проекте.
- Если используется random в тестах, фиксируй seed или делай генерацию воспроизводимой.

## Build and Checks

- Используй `./gradlew`, а не системный `gradle`.
- Сначала запускай самый узкий релевантный тест.
- Полный `./gradlew test` запускай перед коммитом/деплоем или если изменение широкое.
- Перед финальным ответом, если уместно, проверь изменения одной или несколькими командами проекта:
    - `./gradlew test`
    - `./gradlew check`
    - `./gradlew ktlintCheck`
    - `./gradlew jacocoTestReport`

## CLI Tools

Используй инструменты, когда они полезны:

- `rg` — быстрый поиск по содержимому. Предпочитай вместо `grep -R`.
- `fd` — быстрый поиск файлов. Предпочитай вместо `find` для простых задач.
- `jq` — просмотр и обработка JSON.
- `yq` — просмотр и обработка YAML, особенно `docker-compose.yml`, GitHub Actions и конфиги.
- `bat` — просмотр файлов с подсветкой.
- `tree` — быстрый обзор структуры проекта.
- `delta` — удобный просмотр git diff.
- `ast-grep` — структурный поиск и правки кода, если regex через `rg` хрупкий.
- `gh` — GitHub Actions, PR, логи и статусы.
- `docker` / `docker compose` — контейнеры, логи и сервисы.
- `psql` / `pg_isready` — PostgreSQL, если доступ настроен.
- `just` — запуск проектных команд, если есть `justfile`.

## Tool Preferences

- Используй `docker compose`, а не старый `docker-compose`.
- Используй `gh run view --log-failed` для анализа упавшего CI, если `gh` авторизован.
- Для Docker-диагностики сначала используй `docker compose ps` и `docker compose logs`.
- Для PostgreSQL сначала используй read-only проверки и диагностические запросы.

## Docker and Deployment

- Не меняй Dockerfile, docker-compose и deploy-настройки без необходимости.
- При изменениях деплоя учитывай, что сервис работает на VPS.
- Не ломай существующий сценарий запуска контейнеров.
- Не добавляй сервисы в compose без явной пользы.
- Не меняй порты, volumes и env-настройки без запроса.

## Database

- Не делай destructive SQL без подтверждения.
- Не удаляй таблицы, колонки и данные без явной команды.
- Для очередей и конкурентной обработки учитывай блокировки и гонки.
- Если используется PostgreSQL, предпочитай надежные механизмы вроде транзакций и `FOR UPDATE SKIP LOCKED`, если это уже соответствует архитектуре проекта.

## yt-dlp and Files

- Не меняй поведение скачивания без учета ограничений Telegram.
- Учитывай размер файлов до отправки.
- Не ломай обработку аудио и видео режимов.
- Аккуратно работай с именами файлов, расширениями и временными директориями.
- Временные файлы должны очищаться.
- Ошибки внешних процессов должны быть диагностируемыми.

## Telegram Bot Behavior

- Пользовательские сообщения должны быть короткими и понятными.
- Не раскрывай внутренние ошибки пользователю без необходимости.
- Для пользователя показывай нормальный статус или краткую причину ошибки.
- Технические детали оставляй в логах.
- Не меняй команды, меню и тексты без запроса.

## Final Response Format

В конце работы кратко напиши:

- что изменено;
- какие файлы затронуты;
- какие проверки запускались;
- если проверки не запускались — почему.
